[LAV-014] Mostrare bottone "Genera account Stripe" quando IBAN è valorizzato e ID Stripe è vuoto
Tag: MISSING Severity: medium Page/Component: Lavoratori → Cerca Lavoratori → detail lavoratore — tab "Documenti e dati amministrativi", sezione "Dati amministrativi" Effort: M
Sintomo / Cosa succede ora Quando il lavoratore ha un IBAN valorizzato ma non ha ancora un ID account Stripe associato, non c'è modo dall'interfaccia di triggerare la creazione dell'account Stripe per quel lavoratore. L'operazione deve essere fatta manualmente fuori dal backoffice o non è supportata, bloccando il flusso di payroll digitale.
Atteso / Cosa dovrebbe succedere Nella sezione "Dati amministrativi", quando la condizione IBAN ≠ vuoto AND ID account Stripe = vuoto è soddisfatta, mostrare un bottone "Genera account Stripe" (o "Crea account Stripe").
Al click, il bottone triggera la creazione dell'account Stripe via API, popola il campo ID account Stripe sul record lavoratore al successo, e nasconde il bottone (dato che la condizione di visibilità non è più soddisfatta).
Note Condizione di visibilità — pattern componente condizionale: stesso pattern di LAV-004, LAV-007, RIC-027/028/029. La condizione qui è più articolata (AND su due campi invece di un solo stato), ma la logica di rendering condizionale è concettualmente la stessa. Valutare se il componente condizionale riusabile già suggerito nei ticket precedenti supporta condizioni composite AND/OR tra campi diversi — se no, è l'occasione per generalizzarlo.
Backend — endpoint Stripe: la chiamata a Stripe per creare l'account va fatta lato backend (Edge Function dedicata) per non esporre chiavi lato client. Il SWE deve:
Verificare se esiste già un endpoint/funzione per la creazione account Stripe (probabile, dato che il payroll esiste) — in quel caso riusarlo.
Se non esiste, crearlo. L'input tipico è lavoratore_id + IBAN + altri dati KYC richiesti da Stripe (nome, cognome, codice fiscale, indirizzo, data di nascita, ecc. — tutti dati già presenti sul record lavoratore).
Gestione stati UX:
Loading: durante la chiamata Stripe (può richiedere secondi), bottone disabilitato + spinner + messaggio tipo "Creazione account in corso…".
Errore: se Stripe restituisce errore (IBAN non valido, KYC incompleto, dati mancanti, ecc.), mostrare messaggio d'errore chiaro con indicazione del campo/azione necessari. Non nascondere il bottone in caso di errore — il recruiter deve poter ritentare.
Successo: aggiornare il campo ID Stripe con il valore restituito, nascondere il bottone, eventuale toast di conferma.
Rischio da mitigare: la creazione di un account Stripe può avere effetti non-idempotenti lato Stripe (creazione doppia, fee, ecc.). Il SWE deve assicurarsi che il bottone non sia ri-cliccabile durante la chiamata (via loading state) e che il backend gestisca la creazione in modo idempotente (es. controllo preventivo se esiste già un account Stripe associato al lavoratore in Supabase prima di chiamare l'API).
Da chiarire col SWE:
Esiste già un flusso di creazione account Stripe nel sistema attuale? Dove? È accessibile via UI esistente o solo via script/admin?
Quali dati servono a Stripe per creare l'account (requisiti KYC) — sono tutti disponibili sul record lavoratore, o serve chiedere input aggiuntivo al recruiter al momento della creazione?
Il flusso Stripe Connected Accounts / Custom Accounts che usate richiede onboarding del lavoratore (es. link via email)? Se sì, il bottone "Genera account Stripe" potrebbe triggerare l'invio di un onboarding link invece che una creazione immediata — da definire con il SWE.

Il campo motivazione è però già gestito nell'header sticky tramite il dropdown condizionale previsto in LAV-004 (motivazione visibile solo quando stato_lavoratore = "Non idoneo").
